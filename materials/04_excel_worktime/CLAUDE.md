# CLAUDE.md — おまかせ開発の規約

このプロジェクトは Claude Code による「おまかせ開発」で進めます。Claude Code は以下の規約に**必ず**従い、コーディングとテストを担当してください。

## 1. やり取りのしかた
- ユーザーとの対話は日本語で行う。コード内のコメント・docstring も日本語で書く
- ユーザーに選択肢を尋ねるときは、**ポップアップ（AskUserQuestion などの専用UI）を使わない**。チャット内に**番号付きの選択肢テキスト**で質問し、ユーザーの入力を待つ

## 2. 技術スタック
- 言語: Python 3.13
- テストツール: **固定しない**。題材の特性に応じて Claude Code が最適なものを選定・提案する
- 依存ライブラリ: 必要になったら `pip install` で追加する

## 3. 開発の進め方（テスト駆動開発 / TDD）★最重要

次の5ステップを**この順序で必ず実行する**。題材が単純でも省略しない。

1. **仕様書を整える**: ユーザーと対話して仕様を固め、`docs/spec.md` に**ファイルとして保存する**
2. **テスト項目を作る**: 仕様を満たすテスト項目を `docs/test_case.md` に**ファイルとして保存する**
3. **テストプログラムを作る**: テスト項目をテストコードに落とす
4. **RED を確認する**: 実装前にテストを実行し、すべて失敗（RED）することを見せる
5. **GREEN まで実装する（おまかせ自動コーディング）**: テストが全部通るまで、実装・テスト実行・失敗箇所の修正を**自律的に繰り返す**。完了条件は「全テスト通過」。全テスト通過後、その題材で `pip install` した外部パッケージ（pytest を含む）を `requirements.txt` に**必ず**記録する

**停止と自走のルール（重要）:**
- **ステップ1〜4 は、各ステップ完了ごとに必ず停止**し、ユーザーが「次へ」と指示するまで次へ進まない（人が仕様とテストのレールを引く区間）
- **ステップ5 は、ユーザーの実装開始の指示を受けたら停止しない**。テスト全通過まで、途中で質問・確認をせず一気に進める（＝おまかせの本領）

**質問と合図のルール（収録の一貫性のため固定）:**
- ステップ1の仕様確認の質問は、必ず「(1) 出力・挙動の内容 →(2) 入出力・実行方法 →(3) 成果物のファイル名」の順で1バッチにまとめて提示する（題材固有の追加確認はこの後ろに足す）
  - 質問には**全体を通した連番（Q1, Q2, Q3 …）**を振る。`Q1-1`・`Q1-2` のような入れ子番号は使わない（カテゴリ (1)(2)(3) は見出しとしてのみ示す）
  - 回答方法として「**番号を半角スペース区切りで質問の順に並べるだけ（例: `1 1 2 1`）でもよい**」と必ず案内する
  - **(2) 入出力・実行方法 のカテゴリには、必ず「成果物の実行方法」を確認する質問を含める**：`python <ファイル名>.py` で直接実行してデモ出力が出る形（`if __name__ == "__main__"` ブロックを設ける）にするか／import して関数を使う形にするか、をユーザーに選ばせる（Claude が黙って決めない）
- ステップ4（RED 確認）完了時、ユーザーへの実装開始の案内は、必ず「**`実装開始` と入力してください**」と表現する（「実装して」等への言い換えをしない）

**絶対のルール:**
- **テストを書く前に成果物プログラムを実装してはならない**（RED を飛ばさない）
- **テスト工程を省略してはならない**（「簡単だから不要」と判断しない）
- 仕様書・テスト項目は**必ずファイルに保存する**（チャットに書くだけにしない）

## 4. フォルダ・ファイル構成

```
プロジェクト直下/
├── .devcontainer/
│   └── devcontainer.json   ← 開発環境の定義（変更しない）
├── docs/
│   ├── spec.md             ← 仕様書（壁打ちで作成）
│   └── test_case.md        ← テスト項目
├── ◯◯.py                  ← 成果物プログラム（単一Pythonファイル・直下に置く）
├── requirements.txt        ← pip install した物を記録（pytest 含む・GREEN 到達後に必ず作成）
└── CLAUDE.md               ← この規約
```

- 成果物プログラムは**単一の Python ファイル**としてプロジェクト直下に置く
- 仕様書・テスト項目は `docs/` に置く
- テストコードの置き場所は、選定したテストツールの慣例に従う（例：pytest なら `tests/`）

## 5. コーディング規約
- **スタイル**: **Google Python Style Guide**（https://google.github.io/styleguide/pyguide.html）に従う
- **docstring**: **必ず**付ける。**モジュール（ファイル冒頭）・関数・クラスすべてに** Google スタイルの docstring を書く。**1行スクリプトでも、ファイル冒頭のモジュール docstring を省略しない**
- **型ヒント**: 関数の引数・戻り値に型ヒントを付ける
- **コメント**: 日本語で記述する
- **命名**: 関数名・変数名・クラス名などの識別子は、意味が伝わる**英語**にする（例: `run_script`・`result`）。**日本語の識別子は使わない**。ただし `test_` で始まるテスト関数名のみ日本語可（§6）。コメント・docstring は日本語で書く
- **整形**: 空白・行長などの整形は autopep8 に任せてよい（スタイル基準は Google スタイル）

## 6. テスト方針
- 「**テストが全部通る＝完成**」をゴールとする
- テストを通すために**テスト自体を甘くしない**（仕様を満たさない手抜き実装をしない）
- 既存のテストを壊さない（デグレ防止）
- テスト関数名は**日本語**で、何を確認するテストかが一目でわかる名前にしてよい（例: `test_出力文字列が完全一致する`）。**日本語にできるのは `test_` で始まるテスト関数名だけ**。テスト内のヘルパー関数名・変数名は英語にする（§5）

## 7. 禁止事項・注意点
- `docs/spec.md`（仕様書）を勝手に変更しない。変更が必要なときは必ずユーザーに確認する
- `.devcontainer/` の設定ファイルを変更しない
- 仕様にない機能を勝手に追加しない
- **テスト工程を飛ばして実装しない**（再掲・最重要）
- 選択肢の質問に**ポップアップUIを使わない**（チャット内テキストで尋ねる）
